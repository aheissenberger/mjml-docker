## Summary

The server must expose a `POST /v1/render` endpoint that accepts an MJML template
string and returns the compiled HTML output. The behaviour is modelled on the official
`mjml.io` REST API.

## Acceptance Criteria

- `POST /v1/render` accepts `Content-Type: application/json` with body `{ "mjml": "<string>" }`
- Route matching is based on URL pathname so query strings do not break endpoint resolution
- On success (HTTP 200) response body is `{ "html": "<string>", "errors": [] }`
- When the MJML source contains non-fatal issues the response is still HTTP 200 and
  `errors` is a non-empty array of `{ "tagName": string, "message": string, "line": number }` objects
- When `mjml` field is missing or not a string the endpoint returns HTTP 422 with body
  `{ "message": "<human-readable error>" }`
- Request bodies larger than the server limit are rejected with HTTP 413
- Request bodies that cannot be fully received within the configured timeout are rejected with HTTP 408
- Requests exceeding per-client rate limit are rejected with HTTP 429
- When authentication fails (see FR-005) the endpoint returns HTTP 401 before processing
- The endpoint uses the `mjml` npm package for rendering; no shell-out or external HTTP call
- CPU-intensive render/minify work executes off the main request thread so the event loop remains responsive under concurrent traffic
- Render processing uses a bounded worker-pool queue; when saturated the endpoint returns HTTP 503 with a retryable error
- Worker-pool behavior can be tuned via environment variables (`RENDER_WORKERS`, `RENDER_QUEUE_SIZE`, `RENDER_TIMEOUT_MS`)
- Disconnected client requests are cancellation-aware: the server drops pending response work and does not attempt to write to closed sockets
- When a render worker terminates unexpectedly during processing, the in-flight request receives HTTP 500; the pool automatically restarts the crashed worker and resumes processing queued tasks
- Internal rendering/minification failures return HTTP 500 (not validation-status 422)
- Response always uses `Content-Type: application/json`

## Verification

- Automated unit/integration tests in `test/` covering:
  - valid MJML → 200 + HTML
  - MJML with errors → 200 + errors array populated
  - missing body field → 422
  - invalid JSON body → 422
  - oversized body → 413
  - timed-out body read → 408
  - rate limit exceeded → 429
  - saturated render queue → 503
  - saturation test verifies both rejection and subsequent recovery request success
  - wrong auth → 401
  - worker crash during render → 500, and subsequent request → 200 (pool recovery)
  - idle worker crash → pool restart, subsequent request → 200
  - client disconnect mid-render → no server crash, subsequent request → 200

## Notes

Reference: https://mjml.io/api/documentation/
The `mjml` npm package exposes a synchronous `mjml2html(input, options)` function that
returns `{ html, errors }`. No streaming or async rendering is required.
Related architectural decision: ADR-0002.

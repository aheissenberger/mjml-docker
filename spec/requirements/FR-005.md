## Summary

All non-health endpoints must be protected by a static API key. Callers must supply the
key as a Bearer token in the `Authorization` header. The key is configured via an
`API_KEY` environment variable. This follows option B: shared secret for private
network deployments.

## Acceptance Criteria

- Requests to protected endpoints without an `Authorization` header return HTTP 401
  with body `{ "message": "Unauthorized" }`
- Requests with `Authorization: Bearer <wrong-key>` return HTTP 401 with body
  `{ "message": "Unauthorized" }`
- Requests with `Authorization: Bearer <correct-key>` proceed to the handler
- The correct key is the value of the `API_KEY` environment variable
- When `API_KEY` env var is unset the server refuses to start with a clear error message
- Protected endpoints: `GET /` (FR-003) and `POST /v1/render` (FR-004)
- Endpoint protection applies to normalized URL pathname (query string does not alter auth behavior)
- Unprotected endpoint: `GET /health` (FR-002) — no auth required
- All 401 responses use `Content-Type: application/json`

## Verification

- Automated tests in `test/` covering:
  - missing header → 401
  - wrong key → 401
  - correct key → passes through to handler
  - `/health` requires no auth
- Server startup test: process exits non-zero when `API_KEY` is not set

## Notes

The key comparison must use a constant-time equality check to avoid timing attacks.
Node's `node:crypto` `timingSafeEqual` is the preferred implementation.
`API_KEY` must not be logged or included in any response body.
Related architectural decision: ADR-0003.

## Summary

The server should extend `POST /v1/render` to accept an optional `options` object in the JSON
request body. This object is intentionally limited to MJML options that are serializable,
request-scoped, and compatible with the API's existing `{ html, errors }` response contract.
When requested, output minification is applied as a post-processing step to rendered HTML.

## Acceptance Criteria

- `POST /v1/render` accepts either `{ "mjml": "<string>" }` or
  `{ "mjml": "<string>", "options": { ... } }`
- `options` is optional; omitting it preserves the current FR-004 behaviour exactly
- When present, `options` may contain only these keys:
  - `fonts?: Record<string, string>`
  - `keepComments?: boolean`
  - `minify?: boolean`
- Requests containing unknown `options` keys, non-object `options`, or values of the wrong type
  return HTTP 422 with a human-readable validation error
- When `options.minify` is `true`, the server post-processes rendered HTML using a safe
  HTML/CSS minifier configuration
- Rendering remains in-process through the `mjml` npm package and continues to return JSON
- Authentication, route shape, and base success/error response formats remain governed by FR-004 and FR-005

## Verification

- Automated tests in `test/` covering:
  - valid request without `options` preserves existing behaviour
  - valid request with `options.keepComments` is accepted
  - valid request with `options.fonts` is accepted and affects generated font imports as expected
  - valid request with `options.minify=true` returns minified output
  - unknown `options` key returns 422
  - invalid `options` value types return 422

## Notes

Allowed request options were chosen by filtering MJML's option surface to request-safe,
JSON-serializable inputs that do not require host filesystem access and do not change control
flow away from the existing response contract.

Explicitly excluded MJML options and rationale:

- `filePath`, `actualPath`, `mjmlConfigPath`, `useMjmlConfigOptions`: couple requests to server-local
  files, `.mjmlconfig`, `mj-include`, and custom-component resolution on the host filesystem
- `preprocessors`: not possible to pass through a JSON API because MJML expects functions
- `validationLevel`: not exposed because `strict` throws instead of returning `{ html, errors }`,
  while `skip` suppresses warnings and weakens the current FR-004 validation contract
- `beautify`, `minifyOptions`: deprecated upstream; avoid freezing deprecated MJML API
  surface into this service contract. Request-level `options.minify` is supported as a server-side
  post-processing flag and does not forward MJML's deprecated `minify` option.
- `juiceOptions`, `juicePreserveTags`, `skeleton`, `printerSupport`, `ignoreIncludes`: undocumented,
  loosely typed, or specialised enough that they should remain out of scope unless a concrete
  product need appears

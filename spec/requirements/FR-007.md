## Summary

The repository must include a Docker Compose definition that runs the API service with a
container health check and a restart policy. This provides a standard local/runtime orchestration
entrypoint without requiring manual container run arguments.

## Acceptance Criteria

- A Compose file exists at the repository root (`compose.yaml` or `docker-compose.yml`)
- The Compose service builds from the repository `Dockerfile` or references the published image
- The Compose service defines required runtime environment, including `API_KEY`
- The repository includes `.env.example` documenting required and optional runtime environment variables
- The Compose service loads runtime variables from local `.env` automatically
- The Compose service maps the HTTP port so the API is reachable from the host
- A container `healthcheck` is configured and validates the API health endpoint (`GET /health`)
- The `healthcheck` probes the same in-container HTTP port configured by `PORT` so non-default port configurations still transition to healthy
- A non-default restart policy is configured (for example `unless-stopped` or `on-failure`)
- `docker compose up` starts the service and the container transitions to healthy status

## Verification

- Manual: `docker compose up --build` succeeds
- Manual: `docker compose ps` reports the API service as healthy
- Manual: with `PORT=8081`, `docker compose ps` still reports the API service as healthy
- Manual: stopping the service process triggers restart behavior per configured policy
- Manual: updating `.env` values for worker/queue timeout changes container runtime behavior without compose file edits

## Notes

Preferred filename is `compose.yaml` (Compose specification default). The health check should
use tooling available in the image and avoid adding unnecessary runtime dependencies solely for
health probing.
README should include concise dev/prod-style `.env` preset examples for local operation.
